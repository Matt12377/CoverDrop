# 专辑墙与封面工作流性能设计

## 背景与实测证据

用户反馈专辑墙滚动、详情进出、聚合搜索、拖入封面和保存封面均有明显卡顿，并且专辑数量越多越严重。

本次审查同时使用了代码调用链、真实扫描快照和运行中 App 的主线程采样：

- 当前可操作测试库包含 346 张专辑。一次滚动采样中，主线程约 608 ms 落在 `LibraryScanSummaryView.albumGrid` 创建卡片后调用 `displayArtistName` 的链路，继续进入 `AlbumDisplayNameCleaning`、`AlbumNameCleaning` 和现场正则编译。
- 真实“华语”SQLite 快照包含 4,948 张专辑和 69,796 首音轨，平均每张 14.11 首、最大 174 首；其中 4,662 张有封面。
- `CoverPreviews` 当前约 13 GB、22,025 个文件。它是冷缓存磁盘 I/O 的次级风险，但当前证据不支持把全部封面同时驻留内存视为首要根因。
- 现有未提交改动的专项测试通过，证明远程数据复用、JPEG 原样写入和空取消短路的基本行为成立；但这些测试没有覆盖主 actor、SwiftUI 整墙失效、图片并发或大库保存更新。
- 聚合搜索原先通过系统 `.sheet` 打开；主线程采样显示约 257 ms 落在 `SheetBridge.presentSheetIfNecessary`、`NSMoveHelper` 等 AppKit sheet 动画栈，而搜索状态提交本身远低于 1 ms。

## 已确认的根因

### 1. 广域状态发布使整面墙反复求值

`LibraryHomeView`、`LibraryScanSummaryView`、详情页和搜索页共同观察体量很大的 `AppModel`。待保存 URL、保存中集合、提示、扫描结果、快照摘要等任一 `@Published` 变化都可能使封面墙重新求值。

保存成功会依次发布保存中、扫描结果、成功提示、待保存清空、保存结束、后台预览结果和快照摘要。专辑越多，这一放大器越明显。

### 2. 卡片派生数据位于 SwiftUI 热路径

每张卡片分别调用 `displayAlbumName` 和 `displayArtistName`，两次都会重新执行同一组展示名决策。展示名决策遍历音轨标签并进入多轮名称清洗；格式标签还会再次扫描该专辑的全部音轨。

这些数据本来只在专辑记录或名称建议变化时才需要更新，却在滚动和父视图失效时重复计算。

### 3. 图片任务没有完整的背压与取消

本地卡片每次加载创建独立 `Task.detached`，外层 SwiftUI task 取消后底层解码仍可能继续；没有共享并发上限或相同请求合并。一次加载还会为 `thumbnailIdentity` 和 `cachedImage` 重复读取文件身份。

远程预览先请求压缩数据，之后才查解码缓存；`NSImage(data:)` 没有明确下采样。不同 URL 请求没有并发上限，滚动和拖拽大图会竞争网络、CPU 和内存。

### 4. Swift 6 当前设置使部分图片处理返回主 actor

工程启用了默认 `MainActor` 和 `NonisolatedNonsendingByDefault`。从主 actor 调用普通 `nonisolated async` 时，下载后的 ImageIO 校验、`NSImage(data:)`、缓存落盘等同步阶段仍可能继承调用者执行器。

### 5. 详情和保存存在主线程文件系统访问

详情出现后每秒同步执行目录存在性检查；保存前同步解析安全书签并检查专辑目录。NAS 断连、休眠或网络抖动时，单次 `fileExists` 的延迟没有交互上界，足以冻结转场。

### 6. 保存后有多轮全库内存更新和全量快照重写

单次保存会至少两次复制/替换专辑数组：写入完成一次，后台预览完成后再一次。每轮都会更新筛选桶并发布全局状态。

随后后台把全部专辑和全部音轨重新转换为快照，清空 SQLite 表并全量插回。该路径为 `O(专辑数 + 音轨数)`，且当前任务未合并、未严格串行；虽然不直接 await，却会造成保存后的 CPU、内存和磁盘抖动。

### 7. 卡片拖放存在设置后立即清除待保存状态的错误

封面墙卡片接收拖放后先设置 pending，再用 `pendingCoverURL: nil` 打开详情；打开函数随即取消 pending。这既丢失用户拖入结果，又额外发布一次全局变化。

### 8. 聚合搜索的系统 sheet 动画形成固定转场停顿

即使搜索状态和网络请求均不阻塞主线程，SwiftUI `.sheet` 仍会触发 AppKit 原生 sheet 布局与移动动画。该阶段实测约 257 ms，并不随业务代码优化消失，因此需要把搜索改为现有详情工作流容器内的目的地切换。

## 目标

- 约 5,000 张专辑时，普通滚动不执行展示名清洗、同步文件 I/O 或无界图片解码。
- 详情、搜索、拖图、保存等局部状态不得重建整面封面墙。
- 所有网络解析、图片解码/校验/落盘、目录检查和 SQLite 写入明确离开主 actor。
- 图片加载有固定并发上限、相同请求合并和取消语义。
- 保存后只发布一次专辑封面结果；SQLite 快照优先增量更新单张专辑封面。
- 聚合搜索在同一工作流容器内切换，进出不再触发系统 sheet 动画。
- `COVERDROP_DEBUG_LOG=1` 时，终端可实时看到关键阶段耗时、线程、缓存命中、并发量和主线程停顿；关闭时不构造日志上下文。
- 保持拖图先暂存、显式保存、失败保留暂存图、扫描只读和名称清洗只影响派生展示名等产品边界。

## 非目标

- 本轮不全面重写 3,000 多行的详情/搜索文件。
- 本轮不直接改为 `NSCollectionView`；它仅作为定向重构仍不达标时的后备方案。
- 不修改扫描边界、音频标签、音乐文件、Ollama 决策或快照 schema 版本。
- 不把网络下载时间算作 UI 主线程延迟；冷网络必须单独记录，但下载期间 UI 仍应可交互。
- 不在本轮主动删除用户现有的 13 GB 缓存目录；新缓存行为应有界，旧缓存清理由后续兼容策略处理。

## 推荐架构

### 1. 低开销性能诊断

新增统一性能日志组件，提供配对 span 和稳定 operation 名称。启用条件严格为环境变量 `COVERDROP_DEBUG_LOG=1`。

日志覆盖：

- 构建/命中封面墙快照；
- 打开详情、详情首帧、返回封面墙；
- 搜索网络与 HTML/JSON 解析；
- 远程预览下载、缓存命中、下采样解码；
- drag start、provider 读取、预取等待、ImageIO 校验、暂存落盘；
- 目录检查、`cover.jpg` 写入、缩略图生成、内存记录更新；
- SQLite 单专辑增量更新或全量回退；
- 调试模式下超过 50 ms 的主线程响应延迟。

日志关闭时不得创建 UUID、读取时钟或求值上下文字串。热路径不再无条件同步 `print`。

### 2. 预计算卡片数据和 O(1) 快照复用

新增轻量 `AlbumCoverCardPresentation`，包含卡片所需的稳定字段：专辑 ID、路径、展示歌手/专辑名、最多两个格式标签、封面 URL、状态徽标、问题摘要和名称增强状态。

`AlbumScanDisplayIndex` 在建立索引时同时生成 presentation。展示名仍调用现有唯一清洗规则，但每张专辑只计算一次；单张专辑或名称建议变化时只重算对应 ID。

索引按 `revision + filter + normalizedQuery` 缓存完整 `AlbumCoverWallSnapshot`。相同 key 重复读取必须复用同一数组存储，不能每次 body 再把全部 ID 映射为 presentation。查询或真实数据变化时才生成新快照。

### 3. 等价隔离的封面网格

把网格抽成不观察 `AppModel` 的等价子树。输入只包含：

- 已缓存的墙体快照；
- 列数和间距；
- 未分轨选择状态；
- 真正影响卡片的轻量动态状态；
- 动作闭包。

详情 pending、保存 spinner、Finder 状态、搜索页状态和快照提示不进入网格等价键。网格动作可以持有 `AppModel` 引用，但不得通过 `@ObservedObject` 订阅它。

打开详情不再清理 pending；卡片 drop 成功后保留刚暂存的图片。关闭详情只走一次清理路径，并在退出动画逻辑完成后释放选中 ID。

专辑从扫描结果消失时，AppModel 在设置新结果时按 deleted IDs 定向清理 transient 状态，不能恢复 body 内的全库 ID 映射。

### 4. 受限、可取消的本地缩略图管线

新增共享缩略图加载器：

- 默认最大同时解码 4 个；
- 按 `URL + 像素尺寸 + 内容修订号` 合并相同请求；
- 排队任务在所有消费者取消后可移除；已开始的 ImageIO 解码不可强制中断，但数量受限；
- 文件身份最多读取一次，常规卡片更新由 presentation 的内容修订号驱动；
- ImageIO 明确下采样并按真实位图字节计入内存缓存。

SwiftUI `.task` 只 await 结构化请求，不直接创建无界 detached 解码。

### 5. 远程预览、搜索和拖图

远程预览必须先查已解码缓存，再查压缩数据缓存，再发网络请求。解码使用 ImageIO 按卡片尺寸下采样，不使用可能延迟展开像素的裸 `NSImage(data:)`。

不同远程预览最大并发 6 个；原图预取使用 `userInitiated` 优先级并与预览缩略图分开计数。相同 URL 的数据请求继续合并；失败不缓存。

搜索客户端在 detached 边界内执行网络后的 HTML/JSON 解析。聚合搜索和 WKWebView 捕获的 HTTP(S) 图片 URL 都触发原图预取。拖入时优先复用捕获/预取 URL，provider 回退逻辑保持兼容。

远程 ImageIO 校验、暂存目录创建和原子写入整体在后台执行；只有成功/错误状态回到主 actor。

详情与搜索共用一个工作流容器：详情尺寸为 `680 × 520`，搜索尺寸为 `1100 × 700`。`AlbumCoverWorkflowPresentationState` 负责目的地与尺寸；搜索通过显式关闭回到详情，专辑移除也回到详情。详情外层 drop receiver 仅存在于详情分支，搜索页只使用自己的 drop zone。

### 6. 保存链路和增量快照

保存顺序调整为：

1. 后台解析书签和检查专辑目录；
2. 后台写入 `cover.jpg`；JPEG 在完整性校验通过后原样复制，其他格式转换为 JPEG；
3. 后台生成稳定的小尺寸预览；
4. 主 actor 一次性更新专辑记录、筛选索引、成功提示和 pending；
5. 后台增量更新 SQLite 的该专辑 cover 字段。

SQLite store 增加可选的单专辑封面增量能力，并保证同一 store 的写操作串行。旧 JSON 或不支持增量的 store 走“每库串行、FIFO 顺序保留全部更新”的全量回退，禁止并发重写同一文件或丢弃中间状态。

内存索引批量替换时每个筛选桶只取出/修改/写回一次；变化较多时一次重建比逐条 COW 更稳定。

### 7. FSEvents

本轮先记录原始变更路径/flags 和归属判定耗时。精确的内部 `cover.jpg`、专辑目录事件继续抑制。若实测出现祖先目录事件导致 `O(路径数 × 专辑数)`，后续用路径索引处理；在没有 flags 证据前不扩大忽略范围，避免漏掉真实外部变化。

## 数据流

```text
扫描/快照加载
  -> AlbumScanDisplayIndex 建立完整专辑索引
  -> 同时预计算 AlbumCoverCardPresentation
  -> 缓存 filter/query 对应的 AlbumCoverWallSnapshot
  -> EquatableAlbumCoverGrid 只消费轻量快照

拖图
  -> drag start 预取原图
  -> drop provider/捕获 URL
  -> 后台下载或命中缓存
  -> 后台校验和暂存
  -> 主 actor 只发布目标专辑 pending
  -> 打开详情但不清除 pending

保存
  -> 后台目录检查/图片写入/小预览生成
  -> 主 actor 一次发布目标专辑结果
  -> SQLite 单行增量更新
```

## 错误与取消语义

- 图片下载、校验或暂存失败不设置 pending，并沿用现有用户可见错误。
- 保存失败保留 pending，不能发布“已保存”状态。
- 目录不存在时显示“未找到专辑”，并保留待保存图片。
- 离屏缩略图请求取消后不得回写已经复用到其他卡片的 `@State`。
- 预取失败只记录调试日志；真实 drop 仍会重试并显示错误。
- 增量快照失败只影响持久化提示，不回滚已经成功写入的 `cover.jpg` 和 UI 状态；下一次扫描或全量回退可恢复一致性。

## 测试与验收

### 确定性自动化测试

- 5,000 张专辑初始化严格计算 5,000 次 presentation；重复快照读取增加 0 次；替换 k 张只增加 k 次。
- 相同 `revision/filter/query` 返回同一快照存储 token。
- 改变详情 pending/Finder/search 状态不改变网格 render key。
- 卡片 drop 后 pending 保留；普通打开详情不发布无意义 AppModel 变化。
- 详情/搜索目的地决定稳定容器尺寸；关闭搜索或移除专辑回到详情；外层 drop zone 只在详情目的地启用。
- 100 个缩略图请求的峰值并发不超过 4；相同 key 只解码一次；排队请求取消后不执行。
- 远程 decoded cache 命中不调用数据 loader；解码、校验、落盘和目录检查不在主线程。
- JPEG 截断文件不替换现有 `cover.jpg`；完整 JPEG 原样写入。
- SQLite 单专辑更新不删除或重插音轨行；连续更新最终状态为最新值。
- 不支持增量的慢 store 同一音乐库最多一个写任务在途，且 A/B/C 更新按 FIFO 全部执行。

### 运行时验收

- 调试模式结构化日志中，非网络 UI 阶段不得出现超过 50 ms 的主线程停顿。
- 热缓存下详情打开首帧、关闭后墙体可交互、拖图 pending 提交目标均低于 50 ms。
- 保存后的主 actor 内存/索引更新低于 50 ms；本地 JPEG 保存总时延目标低于 150 ms，NAS 与网络耗时单独记录且不得阻塞主线程。
- Time Profiler 的滚动调用树中不再出现 `AlbumDisplayNameCleaning`、`AlbumNameCleaning` 或同步 `FileManager`。
- 聚合搜索打开采样不再出现 `SheetBridge` 或 `NSMoveHelper`，业务状态提交低于 50 ms；网络时延单独记录。
- Animation Hitches/SwiftUI 采样在约 5,000 张合成或真实快照下无大于 50 ms 的 hitch；如仍不达标，再评估 `NSCollectionView` 后备实现。

## 现有未提交改动的处理

保留并补强：

- 远程数据缓存和相同 URL 进行中任务合并；
- 聚合结果原图预取；
- 删除 body 中全库 ID 数组监听；
- 空 pending 取消短路；
- 完整 JPEG 的免重编码写入。

实施前补齐它们缺失的线程、缓存顺序、容量、失败、取消、真实 HTTP 预取和截断 JPEG 测试。不得覆盖或丢弃这些现有工作。

## 后备方案

若完成以上定向修复后，在真实约 5,000 张库上仍出现超过 50 ms 的滚动/详情 hitch，再把仅“封面墙网格”替换为 `NSCollectionView + diffable data source`，复用本设计的 presentation、图片管线、拖放和诊断组件。详情、搜索和 AppModel 无需随之重写。
