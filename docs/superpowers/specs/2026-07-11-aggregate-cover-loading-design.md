# 聚合封面加载与拖拽预取设计

## 目标

加快聚合搜索结果中的缩略图显示，以及把聚合结果拖入封面暂存区后的响应速度；最终写入专辑目录的封面仍使用搜索源提供的大图，不降低尺寸或压缩质量。

## 已确认的问题

- 聚合结果卡片的缩略图在 SwiftUI 视图重新创建时会重新请求并解码，当前没有应用级的解码图片缓存，也不会合并同一 URL 的并发请求。
- 聚合拖拽只提供大图 URL；用户落下时才开始下载大图。以 iTunes 实测为例，100px 预览约 7 KB，而 1200px 大图约 438 KB。现有下载请求还使用 `.reloadIgnoringLocalCacheData`，不会复用此前资源。

## 设计

### 远程图片数据缓存

新增一个 actor，按 URL 缓存短时有效的远程图片数据，并保存同 URL 的进行中任务。它接受由调用方提供的异步加载闭包，首次请求启动加载；同时到达的请求等待同一任务；成功的数据按总字节数和 TTL 受限缓存。失败不缓存。

### 缩略图

`RemoteCoverPreviewLoader` 先查询 `NSCache` 中已经解码的 `NSImage`，未命中时通过上述 actor 下载缩略图数据并解码，再放入 `NSCache`。这样既避免重复网络请求，也避免同一缩略图重复解码。

### 拖拽预取

聚合搜索卡片开始拖拽时，经 `AppModel` 调用 `CoverImageStagingCache` 预取该卡片的 `imageURL`。这一步不设置待保存封面，不改变用户“拖入后才暂存”的流程。

落下后，`stageRemoteImage` 从同一个远程图片数据缓存取大图；如果预取还在进行，等待该请求而不是再发起一条下载；如果已完成，则直接做现有的图片校验和暂存文件写入。

## 约束与错误处理

- 保留现有 HTTP 状态、MIME、20 MB 限制、豆瓣 Referer 和图片可读性校验。
- 缓存只保存已通过网络响应和大小检查的数据；无效图片仍在暂存阶段报错。
- 缓存设置 60 秒 TTL、最大 20 项和 64 MB 总数据量，超出时优先淘汰最早缓存项。
- 取消拖拽可能产生一次未使用的后台下载，但不会写入音乐库、不会创建待保存状态，也不会影响后续保存边界。

## 测试

- 数据缓存：顺序重复请求复用结果、并发重复请求只执行一次加载、过期后重新加载。
- 缩略图：命中已解码图片时不再调用网络加载器。
- 聚合拖拽：开始拖拽触发预取；落下复用预取中的/已完成的大图数据。
