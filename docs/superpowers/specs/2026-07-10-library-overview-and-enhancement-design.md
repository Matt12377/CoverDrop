# 音乐库总览与智能解析范围设计

## 目标

减少封面墙顶部信息拥挤，令统计数量与当前筛选、搜索结果一致，并确保音乐库级智能解析只处理当前音乐库中缺封面的专辑。

## 顶部布局

- 左上区域依次显示音乐库名称、音乐库地址和当前动作内容。
- 删除“当前动作”标题，只保留动作内容，例如“正在智能解析 SACD”或“空闲”。
- 右上区域显示紧凑的智能解析进度，进度条固定为约 `240pt`，不再横跨页面。
- 进度条下方显示 `已完成数/总数`，例如 `93/350`，不重复显示动作内容。
- 没有可显示的智能解析进度时，不占用右上进度区域。

## 当前页面数量

- 顶部只显示一项数量，不再同时排列全部统计项。
- 数量标题跟随当前筛选标签：
  - “全部”显示“专辑数量”。
  - 其他标签使用标签名称，包括“缺封面”“未分轨”“标签异常”“track 音轨”“已有封面”“解析失败”“散落音频”。
- 数量值来自当前页面实际可见结果。
- 搜索关键词生效时，数量同步变为当前筛选页中匹配关键词的结果数。
- “散落音频”页统计匹配的散落音频路径数；其他页面统计匹配的专辑数。

## 智能解析边界

- 音乐库级“智能解析”仅遍历传入 `libraryID` 对应扫描结果中的缺封面专辑。
- 已有封面的专辑不加入音乐库级手动解析队列，也不计入该次进度总数。
- 如果当前音乐库没有缺封面专辑，则结束请求并保持非运行状态，进度为 `0/0`。
- 专辑详情中的单张手动解析保持现有行为，仍允许用户显式解析已有封面的专辑。
- 自动扫描后的智能解析仍保持现有规则，只处理缺封面专辑。

## 数据流

1. `LibraryScanSummaryView` 使用当前筛选后的 `filteredAlbums` 或 `filteredLooseAudioPaths` 计算页面数量。
2. 顶部标题由当前 `AlbumScanResultFilter` 决定，不增加新的持久化状态。
3. `AppModel.requestAlbumNameEnhancement(forLibraryID:)` 先筛选缺封面专辑，再建立队列和进度。
4. 进度 `totalAlbums` 使用实际成功加入队列的缺封面专辑数，避免 UI 与任务数量不一致。

## 错误与状态

- Ollama 未启用、扫描结果不存在等错误继续沿用现有中文提示。
- 解析失败仍记录到单张专辑状态，并通过“解析失败”筛选查看。
- 当前动作内容继续优先显示解析进度、实时刷新消息和失败摘要。
- 页面切换或搜索变化只影响显示数量，不改变扫描结果和解析队列。

## 验证

- 修改 AppModel 单元测试，证明音乐库级解析跳过已有封面专辑。
- 修改进度测试，证明进度总数只包含缺封面专辑。
- 保留单张手动解析已有封面专辑的回归测试。
- 运行 `git diff --check` 和 `zsh Scripts/verify.sh`。
- 运行应用并检查顶部布局，确认进度位于右上、宽度紧凑、下方显示 `xxx/xxx`，动作文字位于音乐库地址下方且没有“当前动作”标题。
