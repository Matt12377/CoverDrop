# 封面墙交互性能与诊断设计

## 背景

当前从封面墙打开专辑详情、打开聚合搜索、拖入封面、保存封面并返回封面墙的链路存在明显卡顿。代码审查表明，封面图片已经通过 `LazyVGrid`、后台缩略图解码和内存缓存加载；更主要的问题是 `LibraryScanSummaryView`、详情页和搜索页共同观察体量较大的 `AppModel`，局部状态变化会使封面墙重新求值。封面墙求值时又会重复执行全库 ID 映射以及基于全部音轨标签的名称清洗，从而放大卡顿。

本设计同时提供可从终端直接观察的结构化耗时日志，先让每个交互阶段可定位，再针对已经确认的重算路径进行隔离和缓存。

## 目标

- 打开详情、拖图、保存及返回过程中，不因无关的 `AppModel` 状态变化重算整个封面墙。
- 专辑展示名、格式标签等卡片派生数据只在源数据变化时计算，不在每次 SwiftUI 求值时重复计算。
- 本地或 NAS 目录检查不阻塞主线程。
- 当进程环境包含 `COVERDROP_DEBUG_LOG=1` 时，在启动应用的终端打印关键操作的开始、结束、耗时和上下文。
- 性能日志关闭时不执行字符串拼接或终端写入，保持正式运行路径的额外开销可以忽略。
- 保持现有扫描边界、名称清洗结果、拖图暂存语义、显式保存语义和保存失败后的待保存图片。

## 非目标

- 本次不新增设置页面或设置开关；以后可将统一诊断开关接入设置。
- 不替换现有 ImageIO 缩略图解码器，不引入第三方图片库。
- 不全面重构超过 3000 行的 `LibraryScanSummaryView.swift`。
- 不修改扫描、Ollama、快照 schema 或音乐文件。

## 方案选择

采用“结构化耗时日志 + 定向状态隔离 + 卡片派生数据缓存”。相比只增加日志，该方案能同时解决已经由调用链确认的问题；相比完全依赖 Instruments，它能满足直接从命令行定位卡顿的需求；相比分页或更换图片库，它修复的是当前状态传播和重复计算的根因。

## 诊断日志

### 开关

诊断日志只在 `ProcessInfo.processInfo.environment["COVERDROP_DEBUG_LOG"] == "1"` 时启用。开关判断集中在统一诊断组件中，调用点不得自行读取环境变量。

### 格式

每个操作使用唯一 span ID 配对开始与结束日志：

```text
[性能] 开始 operation=打开详情 span=<id> albumID=<id> thread=main
[性能] 结束 operation=打开详情 span=<id> duration=18.42ms albumID=<id> thread=main outcome=success
```

日志字段采用稳定的 `key=value` 格式。至少包含：

- `operation`：稳定的中文操作名。
- `span`：一次操作的唯一标识，用于并发日志配对。
- `duration`：结束日志中的毫秒耗时，保留两位小数。
- `thread`：`main` 或 `background`。
- `outcome`：`success`、`failure` 或 `cancelled`。
- 与操作有关的轻量上下文，例如 `albumID`、`albumCount`、`source`；不输出图片二进制、书签或完整音轨列表。

统一组件同时支持同步和异步操作。无论成功、失败或取消，都必须输出对应结束日志。关闭日志时不创建 span ID，不采集时钟，也不构造上下文字串。

### 覆盖边界

记录以下用户可感知阶段：

- 构建封面墙展示数据批次，包括专辑数量。
- 打开详情及详情首帧出现。
- 加载当前详情封面缩略图；封面墙缩略图只记录慢加载和失败，不逐卡打印成功日志。
- 打开聚合搜索及搜索请求。
- 拖图 representation 读取、远程下载/图片验证和暂存。
- 写入 `cover.jpg`、更新内存专辑记录和安排快照更新。
- 关闭详情及封面墙重新可交互。
- 专辑目录存在性检查。

默认慢缩略图阈值为 100 毫秒。批量场景输出汇总，不为几千张卡片逐条输出，以免诊断日志本身制造卡顿。

## 状态和数据流改造

### 封面墙展示数据

`AlbumScanDisplayIndex` 在建立或局部更新索引时生成轻量卡片展示数据，至少包含专辑 ID、展示歌手名、展示专辑名、最多两个格式标签、名称增强状态所需的稳定值和封面 URL。名称清洗仍调用现有唯一规则，但每张专辑只在源专辑或名称建议变化时执行一次。

封面墙按筛选条件取得轻量展示数据，卡片不再分别调用 `displayAlbumName` 和 `displayArtistName`。详情页仍按 ID 获取完整 `AlbumScanRecord`。

### 状态隔离

封面墙容器不因详情专属的待保存 URL、保存中状态、Finder 状态或搜索状态而整体失效。详情通过专辑 ID 和窄化后的详情状态访问所需数据。

打开没有暂存图片的详情时，不再向 `pendingCoverURLsByAlbumID` 写入 `nil`。取消暂存只有在该专辑确实存在暂存项时才发布变化。关闭详情只执行一次状态清理，避免动画回调和显式关闭路径重复清理。

扫描结果或筛选条件真正变化时，封面墙仍正常更新。当前滚动位置和已加载缩略图状态应在打开、关闭详情时保留。

### 文件系统检查

详情出现和定时目录检查在后台执行，只把布尔结果发布回主 actor。同一专辑在已有检查尚未结束时不启动重复检查。保存前仍执行即时存在性检查，并保持“未找到专辑”及保留暂存封面的现有行为。

## 错误处理

- 诊断组件不得改变业务错误；计时包装器透传返回值和原始错误。
- 日志写入失败不能影响用户操作。
- 远程拖图、封面写入和目录检查失败均记录 `outcome=failure` 及经过清理的错误摘要。
- 取消聚合搜索或缩略图任务记录 `outcome=cancelled`，不显示额外 UI 错误。

## 测试与验证

采用测试优先：

- 单元测试验证环境开关、稳定日志格式、成功/失败/取消计时以及关闭状态不求值上下文。
- 单元测试验证一次卡片展示数据构建只计算一次展示名，并在单专辑更新时只更新对应条目。
- AppModel 测试验证打开无暂存图片的详情不会发布待保存状态变化，重复取消为空时不发布变化。
- 文件检查测试验证 I/O 不在主 actor 执行并能正确回传目录消失结果。
- 现有封面保存、拖图、名称清洗、显示索引和实时刷新测试必须继续通过。
- 为约 5,000 张合成专辑增加确定性的计算次数回归测试；墙体重算次数使用测试探针验证，不使用依赖具体机器速度的脆弱毫秒断言。

交付前运行：

```shell
git diff --check
zsh Scripts/verify.sh
```

手工验证使用：

```shell
COVERDROP_DEBUG_LOG=1 open <构建出的 CoverDrop.app>
```

终端日志必须能按同一个 span 串起打开详情、搜索、拖图、保存和返回链路，并明确显示每段耗时。
